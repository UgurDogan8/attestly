Attestly for Confluence

Privacy & Security Policy

Last updated: July 12, 2026 · Published at https://ugurdogan8.github.io/attestly/

1. Overview

Attestly is an Atlassian Forge app for Confluence Cloud that lets teams confirm employees have read and understood important pages (policies, SOPs, security guidelines) and produces an audit trail for compliance purposes (e.g. ISO 27001, SOC 2, HIPAA readiness).

Attestly runs entirely on Atlassian Forge: all compute is Forge FaaS, all data lives in Forge storage (KVS), and the app makes zero requests to any host outside Atlassian's own APIs. There is no external server, no external database, and no third party ever sees Confluence content or confirmation records. All data stays inside your Atlassian cloud site's Forge storage, hosted by Atlassian in the same data residency region as your Confluence site.

What Attestly does not store: display names, email addresses, page content/body text, or any other profile attribute. Names are resolved on demand, at render/export time, directly from Confluence's own user API — never persisted. A name change in Confluence is reflected immediately everywhere in Attestly, and Attestly's own storage contains nothing an attacker could use to build a name↔identity mapping beyond what Confluence itself already exposes to the same viewer.

3. How the App Uses Your Data

  • Confirmation records render the "read and confirmed" status on the page byline, the admin dashboard, and the page drill-down, and feed the CSV/PDF audit export.
  • No comments, no emails, no messages are ever posted by the app. Attestly requests zero write scopes of any kind — it cannot create, edit, or delete Confluence content, comments, or permissions.

4. Data Sharing

We do not sell, rent, or share your data with any third party. The app makes no network calls to anything other than Atlassian's own Confluence REST APIs (via Forge's asUser()/ asApp()) and Forge's own KVS storage — there is no raw network client anywhere in this codebase, and manifest.yml's permissions.external is never populated.

5. Permissions (Scopes) Requested and Why

That is the complete list — five scopes, all read/storage, zero write:* scopes, no read:space, no access to Jira or any admin/user-management API beyond the per-user checks above.

6. No Inbound HTTP Endpoints

Attestly declares zero Forge webtriggers. Every feature, including the audit export, is called through Forge's own authenticated invoke() bridge, running under the requesting user's own session. There is no separate token, shared secret, or standalone URL for anything in this app — a continuously enforced guard in the app's own test suite fails the build if a webtrigger is ever added back.

7. Legal Basis and Retention Rationale

Attestly's core function — proving who acknowledged what, when — is a compliance record. Its value is retention, not deletion-on-request: an organization using Attestly for policy sign-off needs that record to remain intact even after the confirming employee leaves. This is a legitimate-interest basis (GDPR Art. 6(1)(f)): the data subject's own action (clicking confirm) created the record, and its retention serves the same compliance purpose the confirmation was requested for.

8. Account Closure / Erasure

When an Atlassian account is closed or a user is deactivated:

  • The confirmation record is retained — deleting it would destroy the compliance evidence it exists to provide.
  • The name is no longer shown. Since Attestly never stores the name in the first place, a closed account simply fails to resolve: the UI renders [deleted user] or [deactivated] instead of a name, everywhere a name would otherwise appear. The accountId itself — a pseudonymous platform identifier — remains in the record.
  • If a data subject wants their accountId itself removed from historical records, route that request through Atlassian's account-level personal-data tooling; Attestly's stance is that the confirmation event is retained as compliance evidence, consistent with the legitimate-interest basis above.

9. Retention and Uninstall

  • No automatic expiry while installed. Confirmation records persist indefinitely by design. Admins can export the full history at any time.
  • Uninstalling the app soft-deletes all Forge-hosted data, subject to Atlassian's standard 28-day retention window. Reinstalling within that window starts with empty storage — Attestly does not itself hold a separate backup. Recovery of pre-uninstall data requires a support request to Atlassian within 21 days, with the customer's consent.
  • Export before you uninstall if you need to keep the record outside Forge storage — CSV and PDF exports are the supported way to retain evidence beyond the app's own lifecycle.

10. Data Residency

Attestly's storage is Forge KVS, which inherits its residency from the Confluence site's own Forge storage realm — Attestly does not choose or configure a region independently, and does not replicate data to any region or provider outside what Forge itself uses for that site. Customers enrolled in Atlassian's Data Residency program get whatever residency guarantee that program provides for Forge apps on their site.

11. What's Out of Scope for v1 (Disclosed, Not Hidden)

Reminders and re-confirmation-on-edit notifications are deferred to a future release, gated behind a rolling re-consent flow when they ship — v1 never requests any write/comment scope, and status is computed from page versions already visible under read:page:confluence.

12. Children's Data

This app is a workplace compliance tool intended for use by employees within an organization's Confluence site. It is not directed at children and we do not knowingly collect data from children.
